Project Baseline

当前项目不是单纯的 Markdown 编辑器，而是一个“多输出、多入口、多语言”的纯前端文档工作台。

它现在已经同时包含：CodeMirror Markdown 编辑、实时预览、可视化插入编辑器、数学公式、图片粘贴、微信富文本复制、 52pojie BBCode 转换、中英文界面、自动保存、导出 HTML、分栏拖拽和专注模式。后续如果继续扩功能，必须把“改哪个文件、会牵动谁” 管清楚，否则很容易只改了一层，结果预览、复制、导出、可视化插入和国际化不同步。

技术栈

Vite 5 + React 18 + CodeMirror 6 + marked + highlight.js + KaTeX + DOMPurify + html2canvas

运行模式

纯浏览器前端，无后端。文件打开、保存、导出、复制、图片处理全部在浏览器侧完成。

当前最重要的事实

国际化、复制格式、可视化编辑和 Markdown 预览都已经形成多条处理链，新增功能不能只改一个地方。

当前主流程

App 持有主文档状态，Editor 负责输入，Preview 负责渲染，复制和导出都取自预览 DOM。

当前维护风险

很多功能以 DOM 结构为中间层衔接，例如可视化编辑、复制富文本、BBCode 转换。如果你改变预览 HTML 结构，复制和论坛输出很可能也要跟着改。

架构与数据流

这个项目的业务逻辑可以拆成 5 层：入口层、编排层、编辑层、渲染/导出层、国际化/辅助层。

1. 入口层

index.html 加载外部 CSS，src/main.jsx 挂载 React 应用。

2. 编排层

src/App.jsx 统一持有主文档状态、主题、文件名、语言、复制菜单和弹窗状态。

3. 编辑层

Toolbar.jsx、Editor.jsx、VisualEditor.jsx 共同构成输入入口。

4. 渲染层

Preview.jsx 负责 Markdown 转 HTML，inlineStyles.js 与 bbcode.js 负责衍生输出。

为什么 App 是核心

src/App.jsx 同时掌管内容状态、语言状态、保存/打开/导出/复制行为、主布局和弹窗，这意味着顶栏功能和跨模块联动都从这里进。

为什么 Preview 是隐性中枢

复制富文本和 BBCode 都不是直接从 Markdown 文本生成，而是从预览出来的 HTML 再转。因此一旦你修改预览结构或样式语义，复制功能也要一起验。

本地存储与状态

当前项目没有后端数据库，所有持久化都依赖浏览器 localStorage。如果你要扩展状态，一定要先决定它是不是应该被保存，以及 Reset 是否需要清掉它。

图片相关改动指南

“添加图片到项目中”在当前项目里至少有两种完全不同的场景。

场景 A：给编辑器增加或修改文档中的图片能力

  • 主 Markdown 编辑器粘贴图片：改 src/components/Editor.jsx，入口是 src/components/Editor.jsx:38。
  • 图片地址策略：改 src/utils/imageUpload.js。
  • 图片策略设置 UI：改 src/components/Settings.jsx 和 src/utils/i18n.js。
  • 首次提示内容：改 src/components/ImageModeHint.jsx。
  • 如果要让可视化编辑器图片策略与主编辑器保持一致，还要改 src/components/VisualEditor.jsx。

场景 B：给应用 UI 本身加图片资源，例如 Logo、背景图、引导插图

  • 当前项目里还没有 src/assets 或 public 目录，因为 UI 目前几乎没有本地图片资源。
  • 如果图片需要参与打包、在组件里 import，建议新建 src/assets/。
  • 如果图片想通过固定路径访问，建议新建 public/。
  • 然后在对应组件中引入图片，例如顶栏改 src/App.jsx，Guide 面板改 src/components/GuidePanel.jsx，可视化欢迎页改 src/components/VisualEditor.jsx。
  • 尺寸、圆角、阴影、布局放到 src/App.css 里处理。

如果你要接真实图床上传

  • 优先把上传逻辑抽到 src/utils/imageUpload.js，让它根据配置返回远程 URL。
  • src/components/Settings.jsx 需要新增图床配置项，例如 API 地址、token、存储桶。
  • src/App.jsx 需要保存这些配置，必要时新增本地存储 key。
  • src/components/VisualEditor.jsx 也要统一走同一上传接口，否则主编辑器和可视化编辑器行为会分裂。

按钮相关改动指南

增加按钮模块在这个项目里至少分成三类：顶栏按钮、Markdown 工具栏按钮、可视化编辑器按钮。三类入口完全不同。

按钮类型 改动主文件 必须同步的文件 说明
顶栏按钮 src/App.jsx src/utils/i18n.js、必要时 src/App.css 例如 Reset、Guide、Open、Save、Export、Copy、Settings、Theme 都在这里
Markdown 工具栏按钮 src/components/Toolbar.jsx src/utils/i18n.js、必要时 src/components/Preview.jsx 用于往 Markdown 文本中插入语法或打开功能面板
可视化编辑器按钮 src/components/VisualEditor.jsx src/utils/i18n.js、src/App.css 这是独立的一套所见即所得工具栏，不与 Toolbar 共用配置

新增顶栏按钮的标准步骤

  1. 在 src/App.jsx:274 附近的顶栏区域新增按钮 JSX。
  2. 在 App 中新增对应处理函数，放在 handleSave、handleOpen 那一组附近，保持编排集中。
  3. 在 src/utils/i18n.js 的 app 命名空间补中文和英文文案。
  4. 如果按钮需要菜单、弹窗、联动状态，还要在 App 新增 state，并决定是否进 localStorage。
  5. 如果按钮样式和现有 view-btn 不一样，再补 src/App.css。

新增 Markdown 工具栏按钮的标准步骤

  1. 先判断它属于哪一类：包裹选区、行前缀、区块插入、纯文本片段、弹窗型工具。
  2. 在 src/components/Toolbar.jsx:7 的 TOOLBAR_ITEMS 里新增条目，或者仿照表格/公式/可视化按钮新增特殊按钮。
  3. 在 src/utils/i18n.js 的 toolbar 下补上按钮名和 title。
  4. 如果这个按钮引入了新 Markdown 语法，确认 src/components/Preview.jsx 能正确渲染。
  5. 如果这个语法还要支持复制到微信或 52pojie，再同步检查 src/utils/inlineStyles.js 与 src/utils/bbcode.js。

新增可视化编辑器按钮的标准步骤

  1. 在 src/components/VisualEditor.jsx:65 的 VIS_TOOLBAR 增加定义。
  2. 在 src/components/VisualEditor.jsx:85 的 execCmd() 或后续辅助函数里补动作逻辑。
  3. 如果新格式会被插回 Markdown，必须同步修改 src/components/VisualEditor.jsx:7 的 nodeToMd()。
  4. 如果新格式需要激活态高亮，再改 src/components/VisualEditor.jsx:221 的 updateFormats()。
  5. 文案统一补到 src/utils/i18n.js 的 visual.tb 下。

风险与重构建议

这部分不是要求你现在就重构，而是告诉你未来项目一大，就会先从哪里痛起来。

热点 1：App.jsx 过重

src/App.jsx 已经同时管理主状态、文件操作、复制、导出、菜单、弹窗和上下文。后续建议拆出更细的 hooks 或 services。

热点 2：VisualEditor.jsx 复杂度高

它同时承担 intro、工具栏、contentEditable、草稿保存、格式检测、图片粘贴、表格插入、代码块插入和 Markdown 转换。

热点 3：i18n.js 已成大文件

翻译、默认内容、Guide 文档和更新日志都挤在同一个文件。后续建议按模块拆分语言包。

热点 4：App.css 已经像样式仓库

当前样式文件 1122 行，已经包含主应用、Guide、Copy 菜单、Visual Editor 等多个子系统。

当前一个工程优势

项目仍然是纯前端，依赖边界清晰，没有后端耦合，重构成本还可控。现在开始建文档和规范是合适时机。

当前一个技术债

可视化编辑器大量依赖 document.execCommand()。这个 API 是旧式方案，虽然现在能用，但后续复杂块结构编辑很可能会受限。

维护规则

下面这些规则适合直接当成后续协作约束，能明显减少“改了 A 忘了 B”的情况。

  1. 新增任何用户可见文案时，同时补 zh 和 en，以 src/utils/i18n.js 为准。
  2. 新增任何导出、复制、预览可见的格式时，至少检查 4 个地方：Toolbar/VisualEditor、Preview、inlineStyles、bbcode。
  3. 新增任何设置项时，不能只改弹窗；还要补 App 中的默认值、持久化、恢复和 Reset 行为。
  4. 新增任何图片能力时，要先区分是文档内容图片还是应用 UI 资源图片，两者入口不同。
  5. 不要手改 dist，所有运行逻辑都应回到 src 和配置文件中修改。
  6. Guide 面板是用户说明书，功能变化后应同步更新它，否则项目会很快出现代码和说明不一致。
  7. 如果改 Preview HTML 结构，必须回归测试：微信复制、52pojie 复制、HTML 导出、数学公式、代码块、表格。

建议后续建立的目录约定

未来如果继续扩功能，建议逐步引入这些目录：src/features/、src/hooks/、src/assets/、src/styles/、docs/。

手册文件基于当前仓库扫描结果生成。后续若项目结构变化，应同步更新本手册，尤其是常见改动入口和本地存储键两部分。
